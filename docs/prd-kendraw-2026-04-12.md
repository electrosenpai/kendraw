## Executive Summary

Kendraw is an MIT-licensed web-based molecular editor designed as a **drop-in modern successor to ChemDraw**. It targets chemistry students, PhD students, post-docs, academic researchers, teachers, and small labs who cannot afford ChemDraw's ~$4,000/year license but need publication-grade structural drawing, reaction and mechanism editing, RDKit-powered property calculations, and full chemistry-format interoperability.

The product promises **migration without loss** from ChemDraw on the core workflow (drawing, reactions, templates, calculations, import/export) while delivering a radically modern UX (glassmorphism, dark mode, multi-document tabs, auto-save, ChemDraw-compatible shortcuts, real-time property panel) that existing open-source alternatives (Ketcher, MolView) do not match.

The architecture is a hybrid web app: a React 18 + TypeScript frontend embedding a lightweight JS chemistry library for instant feedback, and a FastAPI + Python 3.11 + RDKit backend handling heavy compute (LogP, TPSA, IUPAC name↔structure, canonicalization, format conversion). Kendraw is **100% self-hosted** (docker-compose / `npm install`), with a frontend-only static demo on GitHub Pages — no centralized hosted service, no accounts required, no GDPR exposure.

The MVP target is a tool that lets a URD Abbaye PhD student finish a typical day's work without ever opening ChemDraw. The V1 target is feature parity sufficient to claim full migration coverage. Both releases serve a single design north star: **be the modern, beautiful, free editor that chemists actually want to use.**

## Product Goals

### Business Objectives

Kendraw is an open-source project with no revenue model. Goals are framed as impact, adoption, and ecosystem positioning rather than traditional revenue metrics.

1. **Adopt at URD Abbaye reference lab** — ≥80% of URD Abbaye PhD students and researchers use Kendraw weekly within 6 months of MVP release (measured via direct qualitative feedback, not in-app telemetry).
2. **Achieve ChemDraw functional parity** — ≥95% of the formal ChemDraw parity checklist covered within 12 months of dev start.
3. **Establish public OSS traction** — ≥500 GitHub stars, ≥10 external contributors, ≥3 merged non-trivial PRs from external contributors within 12 months of public release.
4. **Reach the broader chemistry community** — ≥50 forks, ≥3 mentions in chemistry community media, ≥20 detectable self-hosted instances within 18 months.
5. **Validate publication quality** — at least one peer-reviewed paper publishes figures generated by Kendraw within 12 months of MVP.
6. **Submit Kendraw academic paper** — manuscript submitted to JCIM or *Journal of Cheminformatics* within 18 months of MVP.
7. **Embed in open-source ELN ecosystem** — at least one operational integration with Chemotion or eLabFTW within 18–24 months.

## Functional Requirements

Functional Requirements (FRs) define **what** the system does. Each FR is testable and assigned a MoSCoW priority.

- **Must Have (MVP)** — required for `v0.1.0` first public release.
- **Should Have (V1)** — required for `v1.0.0` "ChemDraw parity" release.
- **Could Have / Won't Have** — out of scope for this PRD.

### Drawing Engine — Core

#### FR-001: Interactive 2D Drawing Canvas

**Priority:** Must Have (MVP)

**Description:**
Provide an interactive 2D molecular drawing canvas supporting zoom, pan, and an optional alignment grid. The canvas is the primary editing surface and must respond to all pointer and keyboard interactions without perceptible lag.

**Acceptance Criteria:**
- [ ] User can zoom in/out via mouse wheel, pinch gesture, and keyboard shortcut.
- [ ] User can pan the canvas via middle-click drag, two-finger drag, or space-drag.
- [ ] An optional snap-to grid can be toggled on/off via a UI control and shortcut.
- [ ] Canvas state survives a window resize without distortion or content loss.
- [ ] Pointer events (hover, click, drag) reach the correct atom/bond at all zoom levels.

**Dependencies:** None.

---

#### FR-002: Atom & Element Placement

**Priority:** Must Have (MVP)

**Description:**
Allow placement of any element from the full periodic table on the canvas, with support for custom atom labels and formal charges. The user can change an existing atom's element via the periodic table picker or keyboard shortcut.

**Acceptance Criteria:**
- [ ] Periodic table picker exposes all 118 elements.
- [ ] User can place an atom by selecting an element and clicking on the canvas, or by single-letter keyboard shortcut for common elements (C, N, O, H, S, P, F, Cl, Br, I).
- [ ] Formal charges (-2, -1, +1, +2, etc.) can be assigned to any atom and render correctly.
- [ ] Custom atom labels (e.g., `R`, `R1`, `Ar`, `Et`) can be entered and rendered.
- [ ] Implicit hydrogens are computed and displayed according to standard valence rules.

**Dependencies:** FR-001.

---

#### FR-003: Basic Bond Types

**Priority:** Must Have (MVP)

**Description:**
Support drawing and editing of the bond types required for everyday organic chemistry: single, double, triple, aromatic, wedge, and dash.

**Acceptance Criteria:**
- [ ] Each bond type is selectable via the tool palette and has a keyboard shortcut.
- [ ] Bond order can be cycled in place by repeatedly clicking an existing bond with the bond tool.
- [ ] Wedge and dash bonds correctly indicate stereo direction at the chiral atom.
- [ ] Aromatic bonds render with the standard double-line-with-circle or alternating-double-bond convention.
- [ ] Bond rendering remains correct at all zoom levels and on dark/light backgrounds.

**Dependencies:** FR-001, FR-002.

---

#### FR-004: Predefined Ring Library — MVP Set

**Priority:** Must Have (MVP)

**Description:**
Provide one-click insertion of the most common rings: benzene, cyclopentane, cyclohexane, cyclopropane, furan, pyridine, pyrrole, and thiophene.

**Acceptance Criteria:**
- [ ] All 8 rings are accessible from the tool palette and via keyboard shortcuts matching ChemDraw conventions.
- [ ] Inserted rings are oriented consistently and snap to the canvas grid when grid is enabled.
- [ ] Each ring placement is a single undoable action.
- [ ] A ring can be inserted at the cursor position or by clicking the canvas.

**Dependencies:** FR-001, FR-003.

---

#### FR-005: Quick Carbon Chain Drawing

**Priority:** Must Have (MVP)

**Description:**
Allow rapid drawing of carbon skeletons via click-drag-release, producing a zigzag chain whose length follows the cursor.

**Acceptance Criteria:**
- [ ] Click-drag from an atom (or empty space) creates a chain of bonded carbons that follows the pointer.
- [ ] Chain length is determined by drag distance and rendered with standard zigzag bond angles.
- [ ] Releasing finalizes the chain as a single undoable action.
- [ ] Hovering a chain segment shows the implicit hydrogen count.

**Dependencies:** FR-001, FR-002, FR-003.

---

#### FR-006: Wedge / Dash Stereochemistry Display

**Priority:** Must Have (MVP)

**Description:**
Render wedge and dash bonds visually so that the user can communicate stereochemistry. **Computed R/S labels are V1** (FR-027).

**Acceptance Criteria:**
- [ ] Wedge bonds render as solid filled triangles, dash bonds as horizontal hash sequences, both pointing from the chiral atom outward.
- [ ] Wedge/dash direction can be flipped by clicking the bond again with the same tool.
- [ ] Wedge/dash rendering is preserved across save/load round-trips.

**Dependencies:** FR-003.

---

#### FR-007: Real-time Valence Validation

**Priority:** Must Have (MVP)

**Description:**
Continuously validate atom valences against standard chemistry rules and provide minimal but unambiguous visual feedback when an atom is over- or under-bonded.

**Acceptance Criteria:**
- [ ] Atoms violating standard valence are highlighted (e.g., colored outline) within 100 ms of the change that caused the violation.
- [ ] A tooltip on the highlighted atom explains the violation in plain English.
- [ ] Validation runs entirely on the frontend (no backend round-trip).
- [ ] Validation does not block editing — the user can save and export structures with valence warnings.

**Dependencies:** FR-002, FR-003, FR-018.

### Editing & Manipulation

#### FR-008: Selection Tools — MVP Set

**Priority:** Must Have (MVP)

**Description:**
Allow selection of atoms, bonds, and groups of elements via single-click and rectangle-drag selection.

**Acceptance Criteria:**
- [ ] Single click selects an individual atom or bond.
- [ ] Shift-click adds to or removes from the current selection.
- [ ] Rectangle drag selects all atoms and bonds whose midpoint falls inside the rectangle.
- [ ] A visible selection indicator (highlight, halo) shows what is currently selected.
- [ ] Esc clears the selection.

**Dependencies:** FR-001.

---

#### FR-009: Unlimited Undo / Redo

**Priority:** Must Have (MVP)

**Description:**
Provide unlimited undo and redo across all editing operations within a document session.

**Acceptance Criteria:**
- [ ] Ctrl+Z / Ctrl+Y (or Ctrl+Shift+Z) trigger undo and redo respectively, matching ChemDraw conventions.
- [ ] Undo history is per-document (per tab) and persists for the duration of the document session.
- [ ] No fixed maximum undo depth — bounded only by available memory.
- [ ] Each user action (atom add, bond change, paste, transform, etc.) is a single undoable step.

**Dependencies:** None.

---

#### FR-010: Standard Edit Operations (Clipboard + Transforms)

**Priority:** Must Have (MVP)

**Description:**
Support copy, cut, paste, duplicate, rotate (free + 15°/30°/45°/90° increments), and horizontal/vertical mirroring on the current selection.

**Acceptance Criteria:**
- [ ] Ctrl+C / Ctrl+X / Ctrl+V / Ctrl+D operate on the current selection.
- [ ] Pasted content lands at the cursor position when paste is triggered from a menu, or at the original location when triggered from a shortcut.
- [ ] Free rotation is performed via a rotation handle on the selection bounding box.
- [ ] Snap-to-angle rotation is available via modifier (e.g., Shift) and via menu/shortcut for 15°, 30°, 45°, 90°.
- [ ] Horizontal and vertical mirror operations preserve stereochemistry semantics where chemically meaningful (or warn the user).

**Dependencies:** FR-008.

### Reactions & Mechanisms

#### FR-011: Reaction Arrows

**Priority:** Must Have (MVP)

**Description:**
Provide reaction arrow drawing tools: forward arrow, equilibrium arrow, and reversible (single-headed double) arrow.

**Acceptance Criteria:**
- [ ] All three arrow types are accessible from the tool palette and have keyboard shortcuts.
- [ ] Arrows can be placed by click-drag and resized/rotated after placement.
- [ ] Arrows are rendered with publication-quality stroke widths and arrowhead geometry.
- [ ] Arrows are first-class scene objects (selectable, undoable, exportable).

**Dependencies:** FR-001.

---

#### FR-012: Curly Arrows for Mechanisms

**Priority:** Must Have (MVP)

**Description:**
Provide curly arrow tools for drawing electron-pushing mechanism arrows (single-headed for radical, double-headed for electron-pair). **Non-negotiable for the URD Abbaye PhD persona.** If technical cost becomes prohibitive, this is the *only* MVP item that may slip into earliest V1 — to be decided in tech spec.

**Acceptance Criteria:**
- [ ] Curly arrows can originate at a bond, lone pair, or atom and terminate at a bond, atom, or empty space.
- [ ] Arrow curvature is editable via control points after initial placement.
- [ ] Both single-headed (radical) and double-headed (electron pair) variants are available.
- [ ] Curly arrows render correctly across all export formats.
- [ ] Curly arrows survive save/load round-trips.

**Dependencies:** FR-001, FR-011.

---

#### FR-013: Reaction Condition Annotations

**Priority:** Must Have (MVP)

**Description:**
Allow the user to attach text annotations (reagents, solvents, temperatures, time) above and/or below reaction arrows, with proper alignment and typography.

**Acceptance Criteria:**
- [ ] Clicking an arrow with the text tool spawns annotation slots above and below the arrow.
- [ ] Annotations support basic rich text (subscripts, superscripts, Greek letters via shortcuts or symbol palette).
- [ ] Annotations are anchored to the arrow and follow it when moved or resized.
- [ ] Default typography matches publication conventions (italic for variables, etc.).

**Dependencies:** FR-011.

### Modern Workspace UI

#### FR-014: Glassmorphism UI with Dark/Light Mode

**Priority:** Must Have (MVP)

**Description:**
Deliver the visual identity of Kendraw: a glassmorphism-inspired interface with dark mode as default and an alternate light mode. UI animations must feel fluid (60 fps target on commodity hardware).

**Acceptance Criteria:**
- [ ] App launches in dark mode by default.
- [ ] User can switch to light mode via a clearly visible toggle and via a keyboard shortcut.
- [ ] Mode preference persists across sessions (localStorage or equivalent).
- [ ] Glass surfaces (panels, popovers) use backdrop-blur with sufficient contrast on both modes.
- [ ] All transitions and micro-interactions run at 60 fps on a baseline laptop (i5 / M1 equivalent).

**Dependencies:** None.

---

#### FR-015: Real-time Property Panel

**Priority:** Must Have (MVP)

**Description:**
A side panel continuously displays the current structure's molecular formula, molecular weight, and SMILES — recalculated live as the user draws.

**Acceptance Criteria:**
- [ ] Panel updates within 100 ms of any drawing change for structures up to 100 atoms.
- [ ] Panel updates within 500 ms for structures up to 500 atoms.
- [ ] Panel can be collapsed/expanded.
- [ ] Each property has a one-click "copy to clipboard" affordance.
- [ ] Frontend SMILES generation (FR-018) is the primary computation path; backend RDKit is used only when frontend cannot canonicalize.

**Dependencies:** FR-018, FR-020.

---

#### FR-016: Multi-document Tabs

**Priority:** Must Have (MVP)

**Description:**
Allow the user to keep multiple documents open simultaneously in tab form, each with independent state, undo history, and auto-save.

**Acceptance Criteria:**
- [ ] User can open a new blank document in a new tab via menu and shortcut.
- [ ] Each tab shows the document name and unsaved-changes indicator.
- [ ] Tabs can be reordered by drag.
- [ ] Closing a tab with unsaved changes prompts the user.
- [ ] Switching tabs preserves all in-memory state including selection and undo history.

**Dependencies:** FR-001, FR-009.

---

#### FR-017: Local Auto-save

**Priority:** Must Have (MVP)

**Description:**
All open documents are auto-saved to local browser storage at frequent intervals so that an unexpected close or crash never loses more than a few seconds of work.

**Acceptance Criteria:**
- [ ] Auto-save triggers at most 5 seconds after the most recent edit.
- [ ] Auto-saved state survives a page reload, browser close, or crash.
- [ ] On reload, the user is offered a one-click "restore previous session" with all tabs and undo history intact.
- [ ] Auto-save data is stored entirely client-side (localStorage / IndexedDB) — no backend required.
- [ ] Storage quota errors are surfaced to the user with actionable guidance.

**Dependencies:** FR-016.

---

#### FR-018: Frontend SMILES Parsing & Rendering

**Priority:** Must Have (MVP)

**Description:**
A lightweight JavaScript chemistry library (e.g., OpenChemLib JS, smiles-drawer, RDKit.js WASM — choice TBD in tech spec) parses SMILES strings and renders structures instantly without a backend round-trip.

**Acceptance Criteria:**
- [ ] Pasting or typing a SMILES string into the editor input renders the structure in < 200 ms for molecules up to 100 atoms.
- [ ] Parsing errors are surfaced with a clear, in-context error message.
- [ ] All SMILES syntax features required for organic chemistry (rings, branches, aromaticity, isotopes, stereo descriptors) are supported.
- [ ] Frontend chemistry library is isolated behind an abstraction layer so it can be replaced.

**Dependencies:** None.

#### FR-019: ChemDraw-Compatible Keyboard Shortcuts

**Priority:** Must Have (MVP)

**Description:**
Default keyboard shortcuts mirror ChemDraw's most common bindings so that ChemDraw refugees can become productive immediately. Customization is V1 (FR-035).

**Acceptance Criteria:**
- [ ] At least 30 of the most-used ChemDraw shortcuts are bound by default and documented.
- [ ] A discoverable "shortcuts cheatsheet" is accessible via a `?` shortcut.
- [ ] Shortcuts work consistently across Windows, macOS, and Linux (with Cmd ↔ Ctrl mapping).
- [ ] No shortcut shadows a critical browser shortcut without explicit user opt-in.

**Dependencies:** None.

#### FR-020: Drag & Drop File Import

**Priority:** Must Have (MVP)

**Description:**
The user can drag a chemistry file from the OS file manager onto the Kendraw window to import it into a new tab.

**Acceptance Criteria:**
- [ ] Dropping a `.mol`, `.sdf`, or `.smi` file opens its content in a new tab.
- [ ] Multi-file drops open one tab per file.
- [ ] A drop indicator visually confirms the action.
- [ ] Unsupported file types are rejected with an explanatory toast.

**Dependencies:** FR-016, FR-022.

### Chemistry Compute (Backend)

#### FR-021: RDKit Backend Compute API

**Priority:** Must Have (MVP)

**Description:**
A FastAPI backend exposes a JSON REST API providing the MVP-level chemistry computations that go beyond what the frontend library can do: molecular formula, molecular weight, exact mass, canonical SMILES, InChI, and InChIKey.

**Acceptance Criteria:**
- [ ] All MVP endpoints return correct results for at least 100 reference structures from a curated test set.
- [ ] Each endpoint has a documented request/response schema (OpenAPI / Swagger).
- [ ] Endpoints return errors as structured JSON with stable error codes.
- [ ] Median backend response time is < 300 ms for structures up to 500 atoms on commodity hardware.

**Dependencies:** None.

---

#### FR-022: Format Conversion (MVP Set)

**Priority:** Must Have (MVP)

**Description:**
The backend can convert between Kendraw's internal structure model and the MVP file formats: MOL v2000, SDF, SMILES, and InChI. The frontend can also parse SMILES locally (FR-018) for instant feedback.

**Acceptance Criteria:**
- [ ] Round-trip MOL → Kendraw → MOL preserves all atoms, bonds, charges, and stereo flags for 100% of a curated test set.
- [ ] SMILES round-trip preserves canonical equivalence (after canonicalization) for 100% of the test set.
- [ ] InChI generation matches RDKit's canonical output.
- [ ] All conversion errors are surfaced to the frontend with actionable messages.

**Dependencies:** FR-021.

## Non-Functional Requirements

NFRs define **how** the system performs — quality attributes that must hold across all functional requirements.

---

### NFR-001: Performance — Fluid Editing Up To 500 Atoms

**Category:** Performance
**Priority:** Must Have

**Description:**
The editor must remain interactively fluid (no perceptible lag, all operations < 16 ms frame budget where applicable) for molecules of up to 500 atoms with full editing operations (drawing, selection, transform, undo/redo).

**Acceptance Criteria:**
- [ ] Drawing a new bond on a 500-atom molecule completes within 1 frame (16 ms).
- [ ] Pan/zoom on a 500-atom molecule sustains ≥ 30 fps on a 2020-era laptop (i5 / M1 baseline).
- [ ] Property panel updates within 500 ms after an edit on a 500-atom molecule.
- [ ] Documented benchmarks at 100 / 250 / 500 / 750 atoms are part of the release acceptance procedure.

**Rationale:**
This is the *single* technical promise that, if missed, kills the entire project. POC #1 (architecture phase) is bloquant.

---

### NFR-002: Browser Compatibility

**Category:** Compatibility
**Priority:** Must Have

**Description:**
Kendraw runs correctly on the latest two major versions of Chrome, Firefox, Safari, and Edge on desktop. Tablet/mobile is V2 and not in scope here.

**Acceptance Criteria:**
- [ ] Smoke tests pass on each target browser/version pair as part of CI.
- [ ] No browser-specific feature flags are required for MVP functionality.
- [ ] Critical bugs reported on supported browsers are tracked and fixed before release.

**Rationale:**
A web-first product cannot afford to behave inconsistently across the most common browsers used in academic settings.

---

### NFR-003: License Compliance — MIT Compatibility

**Category:** Legal / Licensing
**Priority:** Must Have

**Description:**
Every dependency, asset, font, and code snippet integrated into Kendraw must be compatible with the MIT license. GPL/LGPL/AGPL dependencies are forbidden in the production build.

**Acceptance Criteria:**
- [ ] CI runs an automated license scan on every PR and blocks merges that introduce incompatible licenses.
- [ ] A `LICENSES.md` or equivalent at repo root inventories all third-party licenses.
- [ ] RDKit (BSD 3-clause) is the canonical reference compatible dependency.

**Rationale:**
The project's MIT license is non-negotiable per the brief. Polluting the dependency graph would force either a license change or invasive surgery later.

---

### NFR-004: Self-Hosted Architecture — No Mandatory Cloud Dependency

**Category:** Architecture / Deployment
**Priority:** Must Have

**Description:**
Kendraw must run end-to-end on a single machine or a single laboratory's infrastructure with no mandatory call to any external service.

**Acceptance Criteria:**
- [ ] A docker-compose deployment runs offline after initial image pull.
- [ ] No required telemetry, account system, or hosted backend.
- [ ] Optional integrations (e.g., Zenodo DOI lookups) are clearly marked as optional and degrade gracefully when offline.

**Rationale:**
Brief: 100% self-hosted, no GDPR exposure, no recurring infra cost, alignment with academic users' expectations.

---

### NFR-005: Publication-Quality Vector Output

**Category:** Output Quality
**Priority:** Must Have

**Description:**
Vector exports (SVG, PDF, EPS) must meet the visual standards of top-tier chemistry journals: clean geometry, no rasterized layers, high-quality typography, correct line weights.

**Acceptance Criteria:**
- [ ] At least 5 reference figures matching JACS/Angewandte style guidelines are produced from Kendraw and validated by URD Abbaye contacts before each major release.
- [ ] Vector outputs open and edit cleanly in Adobe Illustrator and Inkscape.
- [ ] Typography defaults are documented and tunable.

**Rationale:**
"Migration without loss" requires that figures generated by Kendraw be at least as good as ChemDraw's, otherwise the core promise collapses.

### NFR-006: Accessibility Baseline

**Category:** Usability
**Priority:** Should Have

**Description:**
Provide a reasonable accessibility floor: sufficient color contrast (WCAG AA on text), basic keyboard navigation through non-canvas UI, ARIA labels on palette buttons and non-drawing controls. Full WCAG AA conformance is V2+.

**Acceptance Criteria:**
- [ ] Automated contrast scanner reports zero AA failures on text in both dark and light modes.
- [ ] All palette and menu buttons are reachable and activatable via keyboard.
- [ ] All interactive non-drawing controls have ARIA labels.
- [ ] Known accessibility limitations are documented.

**Rationale:**
A molecular editor is intrinsically visual — full WCAG AA on the canvas is unrealistic for MVP — but the surrounding UI can and should meet a reasonable baseline.

## User Flows

The most important user journeys for MVP:

**Flow 1 — Daily molecule drawing (URD Abbaye PhD student):**
Open Kendraw → New tab → Draw structure with palette + shortcuts → Inspect formula/MW/SMILES in property panel → Export PNG → Paste into a Word manuscript → Auto-save persists everything.

**Flow 2 — Mechanism for a teaching handout (chemistry teacher):**
Open Kendraw → Draw reactant → Draw reaction arrow with conditions → Draw product → Add curly arrows for the electron-pushing mechanism → Export SVG → Embed in LaTeX/PowerPoint.

**Flow 3 — Quick SMILES exploration (cheminformatics user):**
Paste SMILES into Kendraw → Frontend renders instantly → Tweak structure → Copy canonical SMILES from property panel → Use elsewhere.

**V1 additional flows** include CDXML round-trip migration, IUPAC name → structure conversion, and template-based fast drawing of biomolecules.

## Assumptions

Inherited from the Product Brief and re-affirmed for the PRD.

- **Web-vs-desktop tolerance** — target users accept a web app and have no absolute requirement for an offline desktop tool. *Validation:* URD Abbaye feedback at MVP.
- **Frontend chemistry feasibility** — a lightweight JS chem lib can cover SMILES parsing and basic rendering with sufficient quality. *Validation:* POC #2.
- **Performance feasibility** — React 18 + SVG/Canvas can sustain fluid editing up to 500 atoms. *Validation:* POC #1 (bloquant).
- **RDKit coverage** — RDKit covers all required calculations and conversions, with OPSIN fallback for IUPAC. *Validation:* RDKit audit in architecture phase.
- **CDXML reverse-engineering tractability** — CDXML can be read/written at sufficient fidelity for real URD Abbaye corpus. *Validation:* POC #3.
- **Community response to launch** — a well-launched MVP will attract stars, feedback, and eventually contributors. *Mitigation:* careful multi-channel launch.
- **Citation compliance** — academic users will honor the citation ask even though MIT does not legally enforce it. *Mitigation:* CITATION.cff, README, splash screen, DOI, EXIF metadata.
- **No blocking patents** — no PerkinElmer (or other) patent blocks Kendraw. *Mitigation:* clean-room implementation, visual differentiation.
- **⚠️ Creator availability** — JB has enough sustained time and motivation over 12–24 months to reach a credible V1 alongside his SRE day job. **This is the project's #1 existential risk.** *Mitigation:* attract contributors early, code structured for appropriability, sustainable cadence over intensity.
